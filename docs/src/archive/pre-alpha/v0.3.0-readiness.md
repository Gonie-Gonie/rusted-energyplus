# v0.3.0 Readiness

Status: ready when `.\scripts\dev.cmd v0.3-verify` passes.

## Release Definition

v0.3.0 locks the first input interpretation contract:

```text
RawModel preservation + TypedModel seed parsing + defaults/enums/references/coverage diagnostics
```

The repository already contains `SimulationModel`, `ModelGraph`, and
`ExecutionPlan` foundations, but those are not the public claim for this
milestone. They become compare/output infrastructure evidence in later
milestones unless explicitly promoted with their own cases and gates.

Supported commands:

```powershell
eplus-rs model inspect <input.epJSON>
eplus-rs model compile <input.epJSON>
```

## Completion Checklist

| Requirement | Evidence |
|---|---|
| RawModel preservation | `raw-model-smoke.ps1` checks version, object counts, tracked/untracked object reporting, and unknown object preservation |
| TypedModel seed parsing | `typed-model-smoke.ps1` compiles declared seed object families |
| default handling | compile output reports `defaults_applied` for typed fixtures |
| enum diagnostics | `invalid-enum.epJSON` emits `InvalidEnumValue` |
| reference diagnostics | `missing-reference.epJSON` emits `MissingReference` |
| duplicate-name diagnostics | `duplicate-normalized-name.epJSON` emits `DuplicateName` after trim/case normalization |
| numeric field diagnostics | `invalid-numeric-field.epJSON` emits `InvalidInteger` and `InvalidFieldType` |
| normalized names and typed IDs | `ep_model::NameMap` and typed IDs are covered by Rust tests |
| coverage contract | compile output distinguishes `typed` and `raw-only` object families |
| oracle epJSON intake | typed smoke compiles EnergyPlus-converted `1ZoneUncontrolled` epJSON |

## Verification Command

```powershell
.\scripts\dev.cmd v0.3-verify
```

## Boundary

This milestone does not claim runtime execution, graph execution, result-store
output, EnergyPlus numerical conformance, heat-balance behavior, HVAC behavior,
or plant behavior. Unsupported objects are preserved and reported as raw-only
unless promoted in a later typed subset.

Unit conversion validation and physical/domain range validation are not v0.3
claims. They should gain negative fixtures only when explicit validators are
implemented.
