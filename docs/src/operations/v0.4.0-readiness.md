```powershell
eplus-rs compare weather-fields <weather.epw> <eplusout.eso>
```

- `Site Outdoor Air Dewpoint Temperature`

The EPW parser also locks these radiation-related source fields for later
promotion:

- horizontal infrared radiation intensity

The corresponding ESO solar/radiation diagnostics are not v0.4 blocking
comparisons. They are future candidates:

- `Site Direct Solar Radiation Rate per Area`
- `Site Diffuse Solar Radiation Rate per Area`
- `Site Solar Altitude Angle`
- `Site Solar Azimuth Angle`

## Timestamp Semantics

Hourly points are EnergyPlus-style hour-ending samples. A run period expands
in output order from the begin date through the end date, inclusive. `hour = 1`
is the first hour-ending sample of the day, and `hour = 24` is the final
hour-ending sample of the day. When a run period omits a calendar year, the
current runtime foundation uses 2013 for deterministic date arithmetic.

The v0.4 comparison smokes align Rust EPW-derived values with the first `N`
EnergyPlus hourly ESO samples for the same weather file and run period. They do
not yet claim daylight-saving, leap-year, subhourly, warmup, design-day, or
environment-boundary conformance.

## Promotion Decision

`schedule_constant_001` and `weather_fields_001` remain smoke cases. Their
gates are non-blocking for compatibility claims, even when the smoke command
prints `status: pass`. No v0.4 case becomes conformance until Rust result
artifacts, declared tolerances, machine-readable summaries, human-readable
reports, and a blocking gate are present.

## Boundary

This milestone does not support full `Schedule:Compact` day-type/calendar
expansion, solar/radiation ESO field comparison, heat-balance conformance, HVAC,
or plant simulation. The report artifacts are baseline-only smoke evidence
until Rust result artifacts, declared tolerances, and blocking gates are added.
