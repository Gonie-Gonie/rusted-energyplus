# v0.2.0 Readiness

Status: ready when `.\scripts\dev.cmd v0.2-verify` passes.

## Release Definition

v0.2.0 locks the first conformance harness contract:

```text
case manifest + suite manifest + output requests + EnergyPlus baseline + baseline-only report skeleton
```

This milestone does not compare Rust simulation output against EnergyPlus.
`conformance_claim` remains `false` for the included fixture cases.

Supported commands:

```powershell
eplus-rs conformance validate-case <case.toml>
eplus-rs conformance baseline <case.toml> <oracle-root> <output-root>
eplus-rs conformance report-skeleton <case.toml> <baseline-case-dir> <report-root>
```

## Completion Checklist

| Requirement | Evidence |
|---|---|
| case metadata schema | `ep_conformance::ConformanceCase` parses and validates manifest fields |
| suite schema | `ep_conformance::ConformanceSuite` validates ordered case paths |
| output request schema | `OutputRequest` requires key, variable, frequency, and class |
| false-claim boundary | schema rejects `conformance_claim = true` without conformance class, tolerances, report, and blocking gate |
| baseline generation | `conformance-baseline-smoke.ps1` stages IDF, converts epJSON, and writes EnergyPlus artifacts |
| baseline-only reports | `conformance-report-smoke.ps1` writes report skeletons from baseline outputs |
| fixture coverage | `schedule_constant_001`, `weather_fields_001`, `surface_geometry_001`, and `zone_temperature_diagnostic_001` are schema-valid |

## Verification Command

```powershell
.\scripts\dev.cmd v0.2-verify
```

## Boundary

This milestone is harness-only. It does not claim Rust-vs-EnergyPlus numerical
agreement, schedule/weather parity, geometry parity, heat-balance behavior,
HVAC behavior, or plant behavior.
